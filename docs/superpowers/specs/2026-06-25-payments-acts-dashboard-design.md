# Дизайн: мини-система учёта оплат, проектов и закрывающих документов

**Дата:** 2026-06-25
**Статус:** утверждён к реализации
**Репозиторий:** https://github.com/Oksnard/B-UP-test

## 1. Цель

Рабочий прототип дашборда, который показывает цепочку:

```
проект / юрлицо → оплаты → этап работ → статус закрывающего документа (акта)
```

Фокус оценки — бизнес-логика, структура данных, связи сущностей, расчёт итогов,
разделение слоёв «данные / логика / отображение», а не визуальная сложность.

## 2. Стек и решения

| Решение | Выбор | Обоснование |
|---|---|---|
| Фреймворк | **Nuxt 3 + Nitro** | Vue во фронте (как просят), полноценный серверный слой (Nitro = REST), нативный деплой на Vercel |
| ORM / БД | **Prisma + SQLite (локально) / Neon Postgres (прод)** | Реальное сохранение статусов актов и локально, и в проде |
| Язык | TypeScript | Типобезопасность сквозь все слои |
| Тесты | Vitest | Юнит (домен) + интеграционные (API) |
| UI / README | русский; код и комментарии — английский | Ревьюер русскоязычный; код — отраслевой стандарт |
| Источник данных | реальный парсинг PDF выписки | `bank_statement_project_data_clean.pdf` (5 стр., счёт ИП Громов А.В.) |

## 3. Доменная модель

```
Counterparty (юрлицо / клиент-плательщик)
  id, name, inn, kpp?, ogrn, bankAccount?, bankName?, contactPerson?, createdAt, updatedAt

Project (проект / направление работ)
  id, name, counterpartyId →Counterparty, status(active|closed), description?, createdAt, updatedAt

Payment (банковская операция по счёту агентства)
  id, date, direction(in|out), amount(Decimal), currency,
  counterpartyId →Counterparty, projectId? →Project,
  purpose(text), serviceStage(enum), expenseCategory?(enum),
  invoiceNumber?, contractNumber?, docNumber?, sourceRef(unique), createdAt, updatedAt

Act (закрывающий документ; ТОЛЬКО для входящих оплат)
  id, paymentId →Payment (unique, 1—1), isSent(bool), sentAt?, isSigned(bool), signedAt?,
  managerComment?, createdAt, updatedAt
  // status НЕ хранится — вычисляется (см. §5)
```

**Связи:** Counterparty 1—N Project; Counterparty 1—N Payment; Project 1—N Payment; Payment 1—1 Act (только `direction = in`).

**Энумы:**
- `serviceStage`: `development | support | ads | seo | content | design | other` — оплаченный этап/тип услуги, выводится из назначения платежа. (Решение: enum-поле на Payment, не отдельная сущность — YAGNI; в README отмечено повышение до `WorkStage` как следующий шаг.)
- `expenseCategory` (для `direction = out`): `tax | subcontractor | fee | other`.
- `direction`: `in` (кредит на счёт агентства = клиентская оплата) / `out` (дебет = налоги/ЕНС/НДФЛ/субподряд).

```

```

## 5. Логика статуса акта (ядро — чистая функция + юнит-тесты)

```
computeActStatus(payment, act, now):
  CLOSED              ← isSent && isSigned                          «закрыт»
  NEEDS_ATTENTION     ← !(isSent && isSigned) && (now - payment.date > 30 дней)  «требует внимания»
  AWAITING_SIGNATURE  ← isSent && !isSigned                         «ожидает подписи»
  NOT_SENT            ← !isSent && !isSigned                        «не отправлен»
```

Порядок проверки: закрыт → требует внимания (просрочка по незакрытому) → ожидает подписи → не отправлен.
Порог `STALE_DAYS = 30` — именованная константа. Источник истины — `isSent/isSigned/date`;
статус выводится, не хранится (исключает рассинхрон).

## 6. REST API (Nitro)

Единый конверт ответа: `{ data, error, meta }`.

| Метод | Путь | Назначение |
|---|---|---|
| GET | `/api/summary?<фильтры>` | сводка; все тоталы из `domain/summary.ts`, уважают фильтры |
| GET | `/api/payments?project=&counterparty=&from=&to=&actStatus=&stage=&direction=&q=` | оплаты с фильтрацией **на бэкенде** |
| GET | `/api/projects` | проекты + агрегаты (сумма, кол-во оплат, закрытых/открытых актов, общий статус) |
| GET | `/api/counterparties` | юрлица + агрегаты |
| GET | `/api/expenses?<фильтры>` | исходящие операции (налоги/субподряд), отдельно от оплат |
| PATCH | `/api/acts/:id` | отметить отправлен/подписан + комментарий — **сохраняется** (zod-валидация) |

**Сводка** (`/api/summary`): общая сумма оплат, кол-во проектов, кол-во оплат,
сумма по закрытым актам, сумма по незакрытым актам, кол-во оплат без отправленного акта,
кол-во оплат «отправлен, но не подписан». Дополнительно: суммарные расходы по категориям.

## 7. Импорт PDF (демонстрация подхода к данным)

`server/import/parseStatement.ts`:
1. Текст PDF через `unpdf` (Node, без Python).
2. Regex-сегментация на операции (по дате проводки + якорям счёта/ИНН/суммы).
3. **Классификация направления** (`classify.ts`): кредит на счёт агентства = `in`;
   дебет (УФК/ЕНС/НДФЛ/налог/исходящий перевод) = `out` с `expenseCategory`.
4. Извлечение полей: дата, плательщик(name/ИНН/ОГРН/счёт/БИК/банк), сумма, № документа,
   № счёта из назначения (`сч. № 742`), № договора (`договор № 418`), ref-код.
5. Вывод `serviceStage` и `project` из назначения (ключевые слова: «сопровождение», «Директ/контекстн»,
   «разработк сайт», «публикаци материал», «объявлени», «SEO», «дизайн»).
6. Идемпотентность по `sourceRef` (ref-код или хэш строки).
7. Acts создаются только для `in`, со **детерминированно-реалистичными** статусами
   (часть не отправлена / ожидает / закрыта; у старых оплат → требует внимания) —
   т.к. в выписке статусов актов нет (явное допущение).

Fallback: закоммиченный нормализованный JSON (`data/normalized.json`) — сидинг воспроизводим
без сетевого доступа к PDF. Реальный PDF — в `data/`.

## 8. Фронтенд (дашборд)

- **Сводка** (карточки сверху): все 7 показателей из §6 + расходы.
- **Фильтры**: проект, юрлицо, период (from/to), статус акта, тип услуги, поиск по назначению/клиенту.
- **Таблица «Оплаты»** (входящие): дата, плательщик, проект, сумма, назначение, этап,
  тогглы «акт отправлен» / «акт подписан», бейдж статуса, комментарий менеджера. PATCH сохраняет.
- **Вкладка «Проекты»**: проект → юрлицо → тоталы → общий статус документооборота.
- **Вкладка «Расходы»**: исходящие (налоги/ЕНС/НДФЛ/субподряд) по категориям.
- Адаптивная вёрстка. Визуал простой, но аккуратный.

## 9. Тестирование, деплой, гигиена

- **Vitest:** домен (`actStatus` — все 4 состояния + границы 30 дней; `summary`; `filters`; `classify`)
  — фокус покрытия ≥80%; интеграционные тесты ключевых API на тестовом SQLite.
- **Deploy:** Vercel + Neon Postgres (`DATABASE_URL`), `prisma migrate deploy` + seed; шаги в README.
- **README.md** (рус): архитектура, ER-диаграмма, логика статусов, допущения, локальный запуск, деплой, API.
- **.gitignore**: node_modules, .nuxt, .output, .env*, *.sqlite, .scratch.

## 10. Ключевые допущения

1. Агентство = ИП Громов А.В. (владелец счёта); клиенты = юрлица из кредитовых операций.
2. Только входящие оплаты порождают акты. Исходящие показаны отдельной вкладкой «Расходы».
3. Проект выводится из назначения платежа (имя проекта / № счёта / связка с клиентом).
4. Статусы актов в выписке отсутствуют → синтезируются детерминированно для демонстрации всех состояний.
5. Этап/тип услуги — enum на Payment (не отдельная сущность) для этого объёма; путь развития описан в README.
